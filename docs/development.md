## Building README.md

```bash
npm run build
```

This runs `scripts/build.js`, which:

1. Reads the YAML front-matter from `src/header.yml`.
2. Reads each macro template from `src/macros/<name>.md` and replaces the
   `{{QUIZ_LOGIC}}` placeholder with the contents of `src/js/quiz-logic.js`
   (the CommonJS export block is stripped so the result is browser-compatible).
3. Reads the documentation body from `src/body.md`.
4. Writes everything into a single `README.md` at the repository root.

## Running the tests

```bash
npm test
```

Jest runs two test suites:

| Suite | What it tests |
|-------|--------------|
| `tests/quiz-logic.test.js` | Pure correctness functions (`isSelectionCorrect`) |
| `tests/build.test.js` | The assembled `README.md` has the expected structure (header, macros, body, docs link) |

Add new unit tests to `tests/quiz-logic.test.js` whenever you add or change a
function in `src/js/quiz-logic.js`.  
Add new build assertions to `tests/build.test.js` whenever you add new
structural requirements to the generated `README.md`.

## How the build works

```
src/header.yml          тФАтФАтФР
src/macros/*.md  тФАтФАтФР      тФВ
src/js/quiz-logic.js тФАтФАтФШ  тФВ  scripts/build.js  тФАтФАтЦ╢  README.md
src/body.md         тФАтФАтФАтФАтФАтФАтФШ
```

### The `{{QUIZ_LOGIC}}` placeholder

Each macro template that contains JavaScript can include the token
`{{QUIZ_LOGIC}}` inside its `<script>` block.  The build script replaces it
with all of the functions from `src/js/quiz-logic.js` at build time, so:

- The browser bundle remains self-contained (no extra `<script src>` needed).
- The same functions are covered by Jest unit tests.

Example location inside a macro template:

```html
<script>
  {{QUIZ_LOGIC}}

  const quizData = { тАж }
  тАж
</script>
```

## Adding or modifying a macro

1. **Create** (or edit) `src/macros/<your-macro-name>.md`.  
   - Put the HTML markup and `<script>` block directly in the file.
   - If your script uses `isSelectionCorrect`, add `{{QUIZ_LOGIC}}` at the top
     of the `<script>` block.
2. **Register** the macro name in the `MACRO_ORDER` array inside
   `scripts/build.js` (macros are written to `README.md` in this order).
3. **Build**: run `npm run build`.
4. **Test**: add assertions in `tests/build.test.js` to verify the new macro
   appears in `README.md`, then run `npm test`.
5. **Document**: add usage examples to `src/body.md`.

## Editing the quiz logic

`src/js/quiz-logic.js` contains the shared, pure correctness functions.  They
are inlined into every macro at build time.

When editing this file:

1. Update the corresponding unit tests in `tests/quiz-logic.test.js`.
2. Run `npm test` to confirm nothing is broken.
3. Run `npm run build` to regenerate `README.md` with the updated logic.

## CI pipeline

The workflow at `.github/workflows/ci.yml` runs on every push:

1. **Install** тАФ `npm ci`
2. **Test** тАФ `npm test` (build fails here if tests are red)
3. **Build** тАФ `npm run build`
4. **Commit** тАФ if `README.md` changed, the bot commits it back to the branch
   with the message `chore: regenerate README.md from sources [skip ci]`.

The commit step uses the `ACTIONS_PAT` secret (a Personal Access Token with
`repo` scope) when available, and falls back to `GITHUB_TOKEN` otherwise.  
If you need the bot to push to a protected branch, configure `ACTIONS_PAT` in
*Settings тЖТ Secrets and variables тЖТ Actions*.
